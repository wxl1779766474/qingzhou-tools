# 使用记录抽屉无闪动设计

## 背景

使用记录抽屉中的“全部记录”和“当前工具”筛选按钮，以及“全部清空”和“取消”按钮，目前都会调用完整抽屉渲染。该渲染通过替换 `historyRoot` 的全部内容，销毁并重新创建遮罩、抽屉、标题、筛选按钮和记录列表。

新创建的遮罩和抽屉会再次执行 160–200ms 的入场动画；被点击的控件也会先失去焦点，再在下一帧聚焦到新节点，因此用户看到整个页面闪动。

## 目标

- 筛选切换立即完成，不播放切换动画。
- 保持遮罩、抽屉和筛选按钮 DOM 节点不变。
- 保持焦点停留在用户点击的原筛选按钮上。
- 只替换记录列表内容，并将新列表滚动到顶部。
- “全部清空”和“取消”只替换顶部清空确认控件，并把焦点移动到新控件。
- 保留“全部清空 → 确认清空 / 取消”的二次确认流程。
- 保留首次打开抽屉时原有的淡入和滑入动画。

## 非目标

- 不改变记录的存储、去重、排序和恢复规则。
- 不调整抽屉的视觉样式、尺寸或移动端布局。
- 不为筛选后的记录列表增加淡入或过渡动画。
- 不改变确认清空后删除记录的结果。
- 不重构删除单条记录和恢复记录的现有交互。

## 方案比较

### 方案一：局部更新抽屉内部控件（采用）

抽屉只在打开时完整挂载一次。筛选切换仅更新两个筛选按钮的激活状态和 `aria-pressed`，然后替换 `.history-content` 内的记录内容。“全部清空”和“取消”仅替换 `.history-clear` 与 `.history-confirm-actions` 这块顶部控件。

优点是彻底避免入场动画重播、无关焦点节点销毁和抽屉布局重建，同时保留首次打开动画。改动范围集中在记录抽屉渲染、筛选与清空确认事件处理。

### 方案二：完整重建但禁用重复动画

筛选时仍重建抽屉，只是不附加入场动画。该方案能减轻视觉闪动，但焦点节点和列表仍会被销毁，滚动位置也会出现瞬时变化。

### 方案三：移除所有抽屉动画

删除遮罩和抽屉的动画声明。改动最少，但首次打开体验也会退化，并且没有解决不必要的 DOM 重建。

## 设计

### 渲染职责

将记录内容生成从完整抽屉渲染中拆出，形成三个职责明确的层次：

1. 完整抽屉渲染只负责打开、关闭以及确实需要改变抽屉结构的操作。
2. 筛选视图更新只负责：
   - 根据 `historyUi.filter` 更新两个筛选按钮的 class 和 `aria-pressed`。
   - 根据当前筛选生成记录分组或空状态。
   - 替换 `.history-content` 的内部内容。
   - 将 `.history-content.scrollTop` 设为 `0`。
3. 清空确认控件更新只负责在“全部清空”和“确认清空 / 取消”之间切换，不改变抽屉其他节点。

### 事件流程

点击筛选按钮后：

1. 读取按钮的 `data-value`。
2. 若值与当前筛选相同，直接返回，不做任何 DOM 更新。
3. 更新 `historyUi.filter`，并取消清空确认状态。
4. 调用筛选视图局部更新函数。
5. 不调用完整抽屉渲染，也不通过 `requestAnimationFrame` 重新聚焦。

由于被点击按钮没有被替换，浏览器会自然保持焦点，键盘用户无需额外处理。

### 清空确认流程

点击“全部清空”后：

1. 将 `historyUi.confirmClear` 设为 `true`。
2. 调用清空确认控件局部更新函数。
3. 下一帧聚焦新创建的“确认清空”按钮。

点击“取消”后：

1. 将 `historyUi.confirmClear` 设为 `false`。
2. 调用清空确认控件局部更新函数。
3. 下一帧聚焦新创建的“全部清空”按钮。

两个流程都不调用完整抽屉渲染，因此遮罩与抽屉的入场动画不会重新播放。由于按钮类型发生变化，清空控件本身允许局部替换并显式恢复焦点。

### 空状态与内容

记录分组、时间标签、摘要及“当前工具还没有记录”等文案继续复用现有规则。文本统计工具仍显示“不保存记录”的专用说明。

## 回归测试

在现有零依赖测试体系中增加抽屉局部更新契约测试，确认：

- 筛选事件调用局部更新函数。
- 筛选事件不调用完整抽屉渲染。
- 筛选事件不替换 `historyRoot.innerHTML` 或 `replaceChildren`。
- 筛选切换不再安排下一帧重新聚焦。
- 局部更新函数更新 `aria-pressed`、激活样式和 `.history-content`。
- “全部清空”和“取消”分支调用清空确认控件局部更新函数。
- 两个清空控件分支都不调用完整抽屉渲染。
- 两个分支仍分别聚焦“确认清空”和“全部清空”，保持键盘可用性。

同时运行现有全部测试，确认历史记录、JSON、二维码、Base64、URL、时间戳、文本统计和构建契约没有回归。

## 发布与验收

构建和测试通过后，将 `site/app.js` 同步到公开 GitHub 仓库。GitHub Pages 部署成功后，验收“全部记录”和“当前工具”双向切换，以及“全部清空”和“取消”双向切换：抽屉与背景保持静止，内容立即更新，焦点与可访问状态正确。
